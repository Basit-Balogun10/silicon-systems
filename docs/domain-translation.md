# Domain Translation Table — HW Concepts ↔ SWE ↔ Plain English

> **Living document.** Add a new row every time a sprint introduces a new concept.
> The AI must update this file whenever it teaches a new term — never let a concept
> go undocumented here.
>
> **Three columns by design:**
>
> - **Hardware / Physical Concept** — the actual term used in the field
> - **Software Engineering Equivalent** — maps it to something an experienced SWE already knows
> - **Noob Plain English** — no jargon, no assumed knowledge; one or two sentences a smart
>   person with zero domain background can grasp immediately

## Digital Logic & RTL

| Hardware / Physical Concept                   | Software Engineering Equivalent                            | Noob Plain English                                                                                                                                                                                          |
| --------------------------------------------- | ---------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Clock / Clock Signal**                      | Event loop heartbeat                                       | A chip does nothing on its own — it needs a metronome ticking millions of times per second. Every tick is one "step" where things can change. Without the clock, the chip is frozen.                        |
| **Clock Cycle**                               | One iteration of an event loop                             | One beat of that metronome. Work only happens _on_ a beat. If your logic isn't done within one beat, you've broken timing and the chip produces garbage.                                                    |
| **Flip-Flop (Register)**                      | A variable that only updates at the end of a function call | A tiny memory cell that stores exactly ONE bit (a 0 or a 1). It holds its value steady all cycle and only updates on the next clock tick — like a variable that refuses to change mid-function.             |
| **Combinational Logic**                       | A pure function with no state                              | Logic that just does math instantly with whatever inputs it currently sees. No memory, no waiting. Output changes the moment input changes — like `f(x) = x * 2`.                                           |
| **Sequential Logic**                          | Stateful code that depends on history                      | Logic that _remembers_ previous values. Needs a clock. Like a class with instance variables — the current output depends on what happened in past cycles too.                                               |
| **Finite State Machine (FSM)**                | A `switch` statement over an `enum` state                  | The "brain" of a hardware controller. It's in one state at a time (e.g., IDLE, SENDING, DONE), and moves between states based on inputs — exactly like a state machine in a game or UI flow.                |
| **State Encoding**                            | Enum representation choice                                 | The way states are represented in bits (for example binary or one-hot). Different encodings trade off area, speed, and debug clarity.                                                                       |
| **Guard Condition**                           | Transition precondition in a workflow                      | A boolean condition that must be true for an FSM transition to occur on this cycle.                                                                                                                         |
| **Bus (e.g. data bus)**                       | A shared network interface / message bus                   | A group of wires that carry data between parts of a chip. Think of it like a highway with lanes — 8 lanes = 8-bit bus = can carry one byte at a time.                                                       |
| **Signal**                                    | A variable                                                 | A named wire. Its value at any moment is either 0 or 1 (or a multi-bit number). Declaring a signal is like declaring a variable — except it exists in physical hardware.                                    |
| **Port**                                      | A function parameter / API endpoint                        | The inputs and outputs of a hardware module. When you plug Module A into Module B, you connect their ports — like calling a function with arguments and getting a return value.                             |
| **Module**                                    | A class or function                                        | The basic building block in hardware design. A module has inputs, outputs, and internal logic. You instantiate modules inside other modules, just like objects.                                             |
| **Parameterized Module**                      | A generic / template class                                 | A module where you can configure its size (e.g., "make this counter 8-bit or 16-bit") at instantiation time, like a `List<T>` in Java or a TypeScript generic.                                              |
| **Counter**                                   | A loop index that persists between iterations              | A hardware block that increases a stored number by 1 on each clock tick. When it reaches the maximum value, it wraps back to 0 and continues.                                                               |
| **Vehicle Request Sensor**                    | Event flag / user action                                   | A detector at the intersection that tells the controller someone is waiting. It is like a button press that asks the system to move forward when it is safe.                                                |
| **Dwell Counter / Phase Timer**               | Timeout / rate-limit window                                | A counter that measures how long the current light phase has been active. It is like a kitchen timer that must finish before the next step is allowed.                                                      |
| **Traffic Light Controller FSM**              | Workflow state machine                                     | The block that keeps one light phase active at a time and decides when to switch to the next one. It is like a traffic manager that will only change the lights after the rules say it is safe.             |
| **Output Decode**                             | View rendering function                                    | Logic that turns internal state into the visible red, yellow, and green lamp pins. It is the final translation step before the outside world sees the result.                                               |
| **Reset Signal**                              | Reinitialize app state                                     | A control signal that forces hardware back to a known safe starting value, like restarting an app and clearing in-memory state.                                                                             |
| **Reset Polarity (Active-High / Active-Low)** | Boolean semantics inversion                                | Reset polarity defines which logic level means "reset is ON." Active-high means 1 triggers reset. Active-low means 0 triggers reset, often named with a trailing \_n (example: reset_n).                    |
| **Enable Signal**                             | Feature flag for update logic                              | A control line that decides whether a block is allowed to update on this cycle; if disabled, state is held unchanged.                                                                                       |
| **Hold Behavior**                             | No-op / preserve previous state                            | When the enable is off, the block keeps the same value instead of changing it. It is like telling a piece of software to return early without mutating anything.                                            |
| **Reset Priority**                            | Fail-safe override / emergency stop                        | If reset and normal work both want attention, reset wins first and forces the block back to a known safe value before anything else can happen.                                                             |
| **Wraparound (Modulo Behavior)**              | Circular buffer index rollover                             | When a fixed-width value reaches its maximum and the next increment returns it to zero, like an array index rolling back to the start.                                                                      |
| **Pipeline**                                  | An async processing pipeline / stream                      | Breaking a long computation into stages so multiple items can be in-flight at once. Stage 1 processes item 1 while Stage 2 processes item 2 — like an assembly line. Increases throughput but adds latency. |
| **Data Path**                                 | The "compute" part of your code                            | The wires and logic that actually move and transform data — adders, multipliers, shifters. The part that does the work.                                                                                     |
| **Control Path**                              | The "orchestration" part of your code                      | The logic that decides _when_ and _how_ the data path operates — FSMs, enable signals, mux selects. Like the conductor of an orchestra.                                                                     |
| **RTL Structure Map**                         | High-level module design sketch                            | A simple block map of where state is stored, where next values are computed, and where control decisions are made. It is the "who does what" blueprint before coding.                                       |
| **always_ff**                                 | State-update function boundary                             | A SystemVerilog block used for clocked state updates only. It helps you keep memory updates disciplined so state changes happen only on clock edges.                                                        |
| **always_comb**                               | Pure function block                                        | A SystemVerilog block used for combinational logic only. It means output is recalculated whenever inputs change, with no memory stored inside that block.                                                   |
| **Nonblocking Assignment (`<=`)**             | Commit-at-end update                                       | Assignment style used in clocked logic so updates take effect together after the edge. This avoids accidental ordering bugs when multiple registers update in the same cycle.                               |

## Timing & Verification

| Hardware / Physical Concept                   | Software Engineering Equivalent                            | Noob Plain English                                                                                                                                                                                                        |
| --------------------------------------------- | ---------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Setup Violation**                           | A data race (write arrives too late)                       | Your logic took too long to compute and the answer wasn't ready before the next clock tick grabbed it. The flip-flop captured garbage. Like a waiter bringing your order after you already left the table.                |
| **Hold Violation**                            | A data race (write changes too fast)                       | Your logic changed its output _too quickly_ after the clock tick — before the flip-flop was done reading. The flip-flop got confused mid-read.                                                                            |
| **fMax (Max Frequency)**                      | Maximum throughput / max safe event loop speed             | The fastest the clock can tick before timing violations appear. Higher fMax = faster chip. Think of it as the chip's speed limit.                                                                                         |
| **Testbench**                                 | Unit test harness                                          | A wrapper you write to "poke" your hardware module with inputs and check that the outputs are correct — exactly like a unit test for a function.                                                                          |
| **DUT (Design Under Test)**                   | System Under Test (SUT)                                    | The module you're currently testing. The testbench feeds it inputs; the DUT is what you're trying to verify works correctly.                                                                                              |
| **Scoreboard (UVM)**                          | A golden reference model + assertion                       | A piece of your testbench that independently computes what the _correct_ output should be, then compares it to what the DUT actually produced. If they differ, the test fails.                                            |
| **Reference Model**                           | An oracle / pure expected-state function                   | A small software copy of the behavior you expect the hardware to follow. The test feeds the same inputs to this model and the DUT, then checks whether both land on the same answer every step.                           |
| **UVM (Universal Verification Methodology)**  | A full testing framework (like Jest or JUnit for hardware) | An industry-standard framework for writing complex hardware tests. It has pre-built components for generating random inputs, monitoring signals, and checking correctness. Required knowledge for professional ASIC jobs. |
| **Cocotb**                                    | pytest for hardware                                        | A Python library that lets you write hardware tests in Python. You control a simulated chip from Python code — `await` a clock edge, drive signals, assert outputs.                                                       |
| **Coverage**                                  | Code coverage                                              | A measure of how much of your hardware's possible behaviour your tests have actually exercised. 100% coverage = you've tested every state, every path, every edge case.                                                   |
| **Verification Scenario**                     | A single named test case                                   | One concrete test story: what inputs you apply, what behavior you expect, and what would count as failure.                                                                                                                |
| **Verification Matrix**                       | A test plan table                                          | A structured checklist that lists all planned scenarios and their expected outcomes so you do not rely on memory or guesswork.                                                                                            |
| **Verification Plan (vPlan)**                 | Full QA strategy document                                  | The top-level verification blueprint for a project. It defines what to test, how to test, coverage goals, and pass criteria. The verification matrix is usually one table inside this larger plan.                        |
| **G0 Checkpoint**                             | Readiness gate before implementation                       | A pre-check stage used to confirm you understand terms and deliverable shape before doing heavier work. If G0 is not clear, coding starts with confusion and rework.                                                      |
| **Cycle Label Consistency**                   | Test step indexing hygiene                                 | The cycle number in your stimulus script must match the cycle number used in expected outcomes and checker timing. If labels drift (for example script ends at C3 but expected says C4), debug becomes misleading.        |
| **Stimulus Pattern**                          | Test input sequence                                        | A step-by-step input script over cycles. Concrete Project-A example: C1 rst=1 en=0, C2 rst=0 en=1, C3 rst=0 en=1, C4 rst=0 en=0, C5 rst=0 en=1. This forces reset, increment, hold, then increment again.                 |
| **Test Intent**                               | Test objective statement                                   | A one-line statement of what a test is supposed to prove. Example: "prove reset has priority over enable at the sampling edge."                                                                                           |
| **Pass Condition**                            | Test success criterion                                     | The exact rule that decides whether a test is marked success. If the rule is true at the required check points, the test passes; if not, it fails.                                                                        |
| **Smoke Test**                                | Fast health check test                                     | A quick sanity check that confirms the design is alive and basic behavior works, before running heavier test suites.                                                                                                      |
| **Directed Test**                             | Deterministic unit test                                    | A hand-crafted test with specific inputs to hit a specific behavior or corner case exactly.                                                                                                                               |
| **Randomized Test**                           | Fuzz-style test                                            | A test that generates varied input patterns automatically to discover edge cases you might not think of manually.                                                                                                         |
| **Checker Hook**                              | Assertion checkpoint / validation tap                      | A specific place in a test where you check if behavior is correct. Example: right after a clock edge, compare expected count vs actual count.                                                                             |
| **RisingEdge (Cocotb)**                       | Await next event tick                                      | A Cocotb wait point that pauses test code until the clock goes from 0 to 1. It is how you align checks to the same timing edges the hardware uses.                                                                        |
| **Assertion**                                 | An `assert` statement                                      | A statement that says "at this moment, this must be true." If it isn't, the simulation immediately flags a failure. In hardware, assertions catch bugs that would otherwise take hours to trace.                          |
| **Invariant**                                 | Always-true contract                                       | A rule that must hold in every valid run, not just one test. If it breaks once, the design is wrong.                                                                                                                      |
| **Coverage Bin**                              | Bucket in analytics metrics                                | A named bucket that counts when a specific behavior happened (for example wrap occurred, reset occurred, hold occurred). Bins prove important behaviors were actually exercised.                                          |
| **Framing Error (UART)**                      | Invalid packet boundary                                    | A receive-side error where frame boundaries are wrong (for example invalid stop-bit timing), meaning the byte framing contract was violated.                                                                              |
| **Mutual Exclusion (One Active Output Rule)** | Single-writer lock rule                                    | A safety rule saying conflicting outputs must never be active at the same time. For traffic lights, Red and Green cannot both be on together.                                                                             |

## Physical Design (ASIC)

| Hardware / Physical Concept                       | Software Engineering Equivalent                              | Noob Plain English                                                                                                                                                                              |
| ------------------------------------------------- | ------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **RTL (Register Transfer Level)**                 | Source code                                                  | Your SystemVerilog design — the human-readable description of what the hardware does. RTL describes _what_ moves _where_ and _when_.                                                            |
| **Synthesis**                                     | Compilation                                                  | The process of turning your RTL (source code) into a list of actual logic gates (like AND, OR, flip-flops). Like compiling Python to bytecode — except the "bytecode" is physical transistors.  |
| **GDSII File**                                    | Final compiled binary / machine code                         | The manufacturing blueprint for a chip. It describes every transistor and wire as geometric shapes on layers of silicon. You send this file to the fab (factory) and they print the chip.       |
| **DRC (Design Rule Check)**                       | Linter / static analysis                                     | An automated check that your chip layout obeys the physical rules of the manufacturing process (wires can't be too thin, too close together, etc.). Violations = the fab can't print your chip. |
| **Floorplanning**                                 | Repository / module folder structure                         | Deciding where each big block of logic goes on the chip's physical area before detailed layout begins. Bad floorplanning = long wires = slow chip.                                              |
| **Standard Cell**                                 | A library function / built-in primitive                      | Pre-designed, pre-characterised basic logic gates (AND, flip-flop, MUX) provided by the chip fab. You don't design these — you assemble your chip _from_ them.                                  |
| **Place and Route (PnR)**                         | Dependency resolution + memory layout                        | The automated process of physically placing each standard cell somewhere on the chip (place) and then drawing the metal wires connecting them (route).                                          |
| **PVT Corners (Process / Voltage / Temperature)** | Environment-based CI testing (prod vs. staging vs. degraded) | Testing your chip under worst-case conditions: slow manufacturing process, low voltage, extreme temperature. A chip that only works in ideal conditions is not a shippable chip.                |
| **STA (Static Timing Analysis)**                  | Performance profiling on the critical path                   | An automated tool that traces every possible signal path in your chip and checks if each one meets timing. It finds your bottleneck (the slowest path) without running the chip.                |
| **Tapeout**                                       | Shipping to production                                       | The point of no return — you send the GDSII file to the fab. Like merging to `main` and deploying to prod, except mistakes cost $50,000–$500,000 to fix.                                        |

## Protocols & Interfaces

| Hardware / Physical Concept     | Software Engineering Equivalent                            | Noob Plain English                                                                                                                                                                                                        |
| ------------------------------- | ---------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **UART**                        | Serial port / simple text socket                           | The simplest way for two chips to talk. One wire sends, one receives. They agree on a speed (baud rate) and send one bit at a time. Like texting — slow but dead simple.                                                  |
| **Baud Rate**                   | API request rate / messages per second                     | The speed of serial communication in symbols per second. In basic UART use, you can treat it as bits per second: higher baud means faster transfer but stricter timing.                                                   |
| **Shift Register**              | Bit queue serializer                                       | A register that shifts bits left or right each cycle so parallel data can be turned into a serial stream, or vice versa.                                                                                                  |
| **SPI**                         | Synchronous RPC call                                       | A faster protocol where a "master" chip controls a clock line and pushes/pulls data from "slave" chips. 4 wires total. Like a function call — the master initiates, the slave responds.                                   |
| **I2C**                         | A shared message bus with device addressing                | Multiple devices share just 2 wires. Each device has an address, and the master calls them by address. Like UDP multicast — everyone hears, but only the addressed device responds.                                       |
| **AXI / AXI-Lite**              | REST API with request/response handshake                   | A standardised bus protocol used inside chips (SoCs) for connecting a CPU to peripherals. AXI-Lite is the simple version (like a basic REST API). AXI4 is the full version with burst transfers.                          |
| **Wishbone**                    | A simpler internal API protocol                            | An open-source, simpler alternative to AXI for connecting blocks inside a chip. Easier to implement but less performant.                                                                                                  |
| **Clock Domain Crossing (CDC)** | Race condition between two async threads                   | When data moves from a circuit running at one clock speed to a circuit running at a _different_ clock speed. Without special handling, bits get corrupted. Like two threads writing to a shared variable without a mutex. |
| **Memory-Mapped Register**      | Hardware peripheral exposed as a struct at a fixed address | A peripheral (UART, GPIO, SPI) where the CPU controls it by reading/writing to specific memory addresses — not by calling functions. It's how firmware talks to hardware.                                                 |
| **DMA (Direct Memory Access)**  | Zero-copy async I/O                                        | A hardware block that moves data from memory to a peripheral (or vice versa) _without the CPU doing the copying_. Like `sendfile()` in Linux — the CPU just starts it, then goes and does something else.                 |

## PCB & Physical Electronics

| Hardware / Physical Concept     | Software Engineering Equivalent                         | Noob Plain English                                                                                                                                                                                           |
| ------------------------------- | ------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **PCB (Printed Circuit Board)** | The physical server / hardware that runs your code      | The green board with copper traces and components you've seen inside every phone and laptop. You design it in software (KiCad); a factory manufactures it.                                                   |
| **Schematic**                   | Architecture diagram / system design doc                | A symbolic drawing showing which components exist and how they connect — without worrying about physical placement yet. It's the _logical_ design, not the physical one.                                     |
| **PCB Layout**                  | Deployment / infra configuration                        | Taking the schematic and physically arranging the components on the board, then routing actual copper traces between them. Where the physical constraints (size, signal integrity) live.                     |
| **Gerber Files**                | Build artifacts / deployment package                    | The manufacturing files you export from KiCad and send to the PCB factory. They define every copper layer, drill hole, and silkscreen exactly.                                                               |
| **BOM (Bill of Materials)**     | `package.json` / `requirements.txt`                     | The list of every component you need to buy to assemble your board — part numbers, quantities, suppliers.                                                                                                    |
| **Decoupling Capacitor**        | A local cache / buffer                                  | A tiny capacitor placed right next to every chip on a PCB. It acts as a local energy reservoir, smoothing out sudden current spikes so the chip's voltage stays stable. Almost every chip needs them.        |
| **Trace**                       | A wire / network cable                                  | A copper path on a PCB that connects two points. For low-speed signals it's just a wire. For high-speed signals (USB, Ethernet) it must be a specific width to control impedance.                            |
| **Impedance Control**           | Protocol negotiation / impedance matching in networking | High-speed signals reflect back if the wire's characteristic impedance doesn't match the chip's output impedance. Like a network cable that must be 75Ω — use the wrong cable and signal integrity degrades. |
| **Ground Plane**                | Global shared return path / common ground reference     | A solid copper layer dedicated to ground (0V). Provides a clean, low-impedance return path for all signals. Without it, high-speed designs fail from noise.                                                  |
| **Power Plane**                 | A dedicated VCC rail layer                              | Same idea — a whole copper layer dedicated to the power supply voltage. Reduces resistance and keeps voltage stable across the board.                                                                        |

## RF & Analog

| Hardware / Physical Concept      | Software Engineering Equivalent                              | Noob Plain English                                                                                                                                                                                                    |
| -------------------------------- | ------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **RF (Radio Frequency)**         | Wireless networking layer                                    | Signals that oscillate fast enough to radiate as radio waves. Wi-Fi, Bluetooth, 5G — all RF. At these frequencies, normal wire = antenna (good and bad).                                                              |
| **Antenna**                      | A network interface card (NIC)                               | A physical structure that converts electrical signals into radio waves (transmit) or radio waves into electrical signals (receive). Shape and size determine what frequencies it works at.                            |
| **Impedance Matching (50Ω)**     | Agreeing on a protocol / API contract                        | Both the chip and the antenna must "expect" the same signal level. If they don't match, power reflects back instead of radiating. 50Ω is the universal agreed standard for RF systems.                                |
| **LNA (Low Noise Amplifier)**    | A signal pre-processor / noise filter before deserialization | The first thing a received radio signal hits. It amplifies the tiny incoming signal while adding as little noise as possible. Critical — noise added here contaminates everything downstream.                         |
| **Modulation (QPSK, QAM)**       | Serialisation / encoding format                              | How you encode digital bits (1s and 0s) onto a radio wave. QPSK encodes 2 bits per symbol. QAM encodes more. Like choosing between JSON and Protobuf — more efficient encoding = more data per transmission.          |
| **SDR (Software Defined Radio)** | A programmable NIC / configurable network stack              | A radio where the modulation, filtering, and protocol logic is implemented in software (or FPGA) rather than fixed hardware. Extremely flexible — same hardware can do Wi-Fi, FM radio, 5G, all by changing the code. |

## Power Electronics & Control

| Hardware / Physical Concept | Software Engineering Equivalent                   | Noob Plain English                                                                                                                                                                                                                              |
| --------------------------- | ------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Buck Converter**          | A DC voltage step-down regulator                  | A circuit that efficiently steps down voltage (e.g., 12V → 5V) using a rapidly switching transistor, inductor, and capacitor. Like a transformer but for DC. Much more efficient than a resistor.                                               |
| **Boost Converter**         | A DC voltage step-up regulator                    | Same idea but steps voltage _up_ (e.g., 3.3V → 12V). Your phone charger uses one of these.                                                                                                                                                      |
| **MOSFET**                  | A software-controlled switch (transistor)         | A transistor used as an electronic switch. A tiny voltage on the "gate" pin turns the switch on or off. At high frequencies (100kHz+), it switches millions of times per second to control power flow.                                          |
| **PID Controller**          | A feedback control loop                           | An algorithm that continuously measures the difference between where you are and where you want to be (error), and adjusts its output to minimise that error. Used in motor control, power supply regulation, drone stabilisation — everywhere. |
| **Dead-Time Insertion**     | Mutex / critical section around a shared resource | A tiny intentional delay inserted between turning one transistor OFF and another ON in an H-bridge. Without it, both transistors briefly conduct simultaneously → a dead short circuit → instant destruction.                                   |
| **H-Bridge**                | A bidirectional motor driver circuit              | A circuit using 4 transistors arranged in an "H" shape that can push current through a motor in either direction. How every DC motor controller works.                                                                                          |

## Machine Learning & Security

| Hardware / Physical Concept        | Software Engineering Equivalent                           | Noob Plain English                                                                                                                                                                                                                             |
| ---------------------------------- | --------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Quantization (Float32 → Int8)**  | Model compression / switching from `double` to `byte`     | Reducing the precision of a neural network's numbers from 32-bit floats to 8-bit integers. The model gets ~4x smaller and runs ~4x faster on hardware — with only a small accuracy loss.                                                       |
| **Hardware Accelerator**           | A GPU / specialised compute worker                        | A custom hardware block (on FPGA or ASIC) designed to do one type of computation extremely fast (e.g., matrix multiply for ML). Like offloading to a GPU, but you designed the "GPU" yourself.                                                 |
| **Side-Channel Attack**            | Inferring secrets from observable system behaviour        | Extracting a secret key not by breaking the math, but by watching _physical_ side effects — like how long an operation takes, or how much power the chip draws. The power trace of an AES encryption leaks the key if you know how to read it. |
| **Masking (Side-Channel Defense)** | Adding noise / dummy operations to prevent timing attacks | Modifying your hardware to add random dummy operations that scramble the power signature. An attacker can no longer correlate power consumption to the secret value.                                                                           |
